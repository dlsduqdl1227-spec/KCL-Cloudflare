### Stage96 KCR 표시 보완

- 강도는 내부적으로 기존 1~7 값을 유지하지만 평가·검수·디브리핑 화면에서는 `매우 약함`부터 `매우 강함`까지 문구로만 표시한다.
- KCR 검수 스마트태그를 평가 화면과 동일한 상위 카테고리 → 하위 카테고리 → 세부 태그 순서로 변경했다.
- 스마트태그 저장값과 기존 평가 데이터, 점수 계산식은 변경하지 않았다.

### Stage97 전 대회 스마트태그 UI 통일

- 스마트태그를 사용하는 KBC, MOB, KCAC, KCR, IKRC의 평가·검수 화면을 같은 계층 탐색 방식으로 통일했다.
- 상위 카테고리를 누르면 해당 하위 카테고리만 열리고, 하위 카테고리를 누르면 세부 태그가 표시된다.
- 기존 선택 태그가 있으면 해당 경로를 자동으로 다시 열며 다른 카테고리의 하위 영역은 숨긴다.
- `긍정/보완`은 탐색용 카테고리로만 사용하고 저장 태그에는 추가하지 않는다. 기존 태그 문자열과 직접 입력 태그 형식은 유지한다.
- MOC와 KTCC는 정답 선택형 대회로 스마트태그 데이터가 없으므로 변경 대상에서 제외했다.
- KBC 서비스 스마트태그 검수 제목에 정규식 문자열이 노출되던 기존 표시 오류를 수정했다.

### Stage98 최종 스마트태그 표시

- Stage98에서는 상위·중간 카테고리를 탐색 전용으로 처리했으나, `과일같은`처럼 하위 특성을 확정하기 어려운 경우 상위 카테고리 자체를 기록할 수 없다는 운영상 누락이 확인됐다.
- 이 동작은 아래 Stage100에서 상위·중간·최종 중 현재 판단 가능한 가장 구체적인 한 단계만 저장하도록 대체했다.

### Stage99 배포 최적화 및 안정성 보완

- 평가와 디브리핑 HTML에 각각 중복 삽입된 7개 대회 로고를 내용 해시 PNG 자산으로 분리했다.
- 평가 HTML은 1,199,462바이트에서 837,365바이트로, 디브리핑 HTML은 485,926바이트에서 123,829바이트로 줄었다.
- 해시 로고는 `Cache-Control: public, max-age=31536000, immutable`로 재사용하고 HTML·API shim·등록 템플릿은 계속 `no-store`로 유지한다.
- 로컬·CI 실행 버전이 설치 시점에 따라 달라지지 않도록 Wrangler를 검증 버전 `4.110.0`으로 고정했다.
- 점수 목록 정렬, 중복 제출 조회, OTP 조회, 보조 세션 조회용 D1 인덱스 4개와 `0005_performance_indexes.sql`을 추가했다.
- 디브리핑 비공개 상태에서는 OTP 문자를 발송하지 않고 기존 OTP도 소모하지 않도록 처리 순서를 수정했다.
- 대회별 평가식, 심사 역할, 라운드 구조, 스마트태그 저장 방식은 변경하지 않았다.

### Stage100 스마트태그 모든 계층 선택

- KCR, KBC, MOB, IKRC의 평가 화면과 KCR·KBC·MOB·IKRC·KCAC 공통 검수 화면에서 의미 있는 1차·2차·최종 태그를 모두 선택할 수 있게 변경했다.
- 같은 경로에서는 `과일같은 → 감귤류 → 오렌지`처럼 하위 단계로 구체화할 때 이전 상위값을 교체하고, 다시 상위 단계를 누르면 해당 하위값을 제거한 뒤 상위값으로 낮춰 기록한다.
- 서로 다른 경로의 태그는 그대로 함께 유지하며 최대 5개 제한을 적용한다. KBC에서 누락돼 있던 최대 5개 제한도 같은 공통 기준으로 보강했다.
- `긍정/보완` 같은 평가 방향 구분은 실제 향미·특성값이 아니므로 기존처럼 탐색 전용으로 유지한다.
- 선택 칩에는 현재 경로에서 최종 확정된 한 값만 표시한다. 점수 계산, 심사 역할, 제출·검수 저장 API와 DB 스키마는 변경하지 않았다.

### Stage101 MOB 자동 코멘트 복구

- Stage51 이후 MOB 중복 UI를 정리하는 과정에서 `mob-comment-section` 전체가 숨겨져 자동 코멘트 버튼과 최종 코멘트 입력란까지 사라진 원인을 확인했다.
- MOB 공통 코멘트 영역을 다시 표시하고, 항목별 스마트태그와 중복되던 별도 참조·태그 영역만 숨겨 모바일 화면의 중복은 유지하지 않았다.
- 코멘트 생성기는 MOB 평가 항목의 실제 1~5점 척도를 사용하도록 수정했다. 1.0, 3.0, 5.0 경계 문구를 API로 검증했다.
- 센서리·테크니컬 계정의 평가 항목, 제출 데이터, 본인 검수 목록은 역할별로 계속 분리된다. 점수 계산·시간감점·순위·초기화 함수는 Stage100과 동일하다.

### Stage102 Android 모바일 포커스 안정화

- `iOS용`으로 작성된 320ms 지연 `scrollIntoView()`가 Android를 포함한 모든 브라우저에서 실행되고, 사용자가 점수를 누른 뒤에도 이전 이름 입력칸을 다시 화면 중앙으로 이동시킬 수 있던 원인을 수정했다.
- 지연 스크롤은 실제 iPhone/iPad에서 현재 포커스가 유지되는 입력칸에만 적용한다. 입력칸이 바뀌거나 포커스를 잃으면 예약된 스크롤을 취소한다.
- 평가·검수 화면의 점수 슬라이더, 선택 버튼, 체크 항목을 누르면 기존 텍스트 입력 포커스를 먼저 해제해 Android 가상 키보드와 커서가 이전 위치로 화면을 되돌리지 않도록 했다.
- 일반 텍스트 입력 중에는 포커스를 유지하며, 점수·스마트태그·코멘트·저장값은 변경하지 않는다.

## 4. 남은 버그 및 운영 제한

현재 로컬 재현 기준으로 순위나 저장을 잘못 계산하는 미수정 코드 버그는 확인되지 않았다. 아래는 현장 투입 전 반드시 이해해야 할 운영 제한이다.

1. **IKRC Technical Time**: 장비 이상 접수, 승인/불승인, 타이머 정지·재개·재시연을 자동 처리하는 전용 기능은 없다. 외부 공식 기록지를 사용할지 운영진 확정이 필요하다. 외부 기록이 없다면 IKRC 운영 전 기능 추가가 필수다.
2. **KCR 정규화**: 시스템은 입력·검수된 `정규화점수/공식점수/순위반영점수`를 순위에 우선 사용하지만, 정규화 공식을 자동 산출하지 않는다. 공식 산식과 입력 책임자를 확정해야 한다.
3. **초기화 범위**: 전체 대회 초기화는 차단되어 있고 대회별 초기화만 가능하다. 라운드 한 개만 초기화하는 기능은 제공하지 않으므로 실제 점수가 들어간 뒤에는 초기화 버튼을 사용하지 말고 관리자 검수에서 개별 수정/삭제해야 한다.
4. **PDF**: 서버가 PDF 파일을 생성하는 방식이 아니라 브라우저 인쇄 창에서 `PDF로 저장`을 선택하는 방식이다. 행사 노트북의 인쇄/PDF 저장 기능을 사전 확인해야 한다.
5. **KCAC 미디어**: 영상 파일 직접 업로드는 용량 안정성을 위해 지원하지 않고 제출 URL을 저장한다. 평가 사진은 모바일에서 압축 저장한다. 실제 영상 링크 권한과 사진 촬영/앨범 권한을 행사 기기로 확인해야 한다.
6. **엑셀 라이브러리**: 브라우저가 XLSX CDN을 불러오지 못하면 호환 `.xls` 방식으로 내려받는다. 행사 네트워크에서 실제 `.xlsx` 다운로드를 확인해야 한다.
7. **원격 환경**: 이번 검수는 로컬 Cloudflare Pages/D1에서 수행했다. 운영 중인 원격 D1과 SOLAPI 키를 사용한 실검증은 수행하지 않았다.

- Stage101 실제 화면: MOB 센서리 계정은 센서리 항목과 자동 코멘트만, 테크니컬 계정은 테크니컬 항목과 자동 코멘트만 표시됨을 확인했다. 각 계정에서 코멘트 선택지 3개 생성, 선택 후 직접 수정 안내를 확인했다.
- Stage101 역할·검수 통합: 로컬 D1에서 센서리 18점과 테크니컬 9점을 각각 제출했다. 각 일반 심사위원의 검수 목록은 본인 제출 1건만 반환했고 반대 역할 점수 필드는 비어 있었으며, 두 건 모두 검수완료 전환에 성공했다.
- Stage101 코멘트 점수 경계: 테크니컬·센서리 5.0은 `매우 선명하고 완성도 높은`, 3.0은 `기준을 충족하는`, 1.0은 `개선이 필요한` 문구로 생성됨을 API에서 확인했다.
- Stage101 모바일: 390×844에서 코멘트 영역과 자동 생성 버튼 폭이 화면 안에 들어오며 문서 가로 넘침이 없었다.
- Stage101 변경 격리: Stage100 대비 변경 파일은 `public/assessment/index.html`, `functions/api/rpc.js`, 본 보고서 3개다. MOB 제출·점수·시간감점·순위·검수 핵심 함수 14개는 Stage100과 동일했고 JavaScript 및 평가 HTML 인라인 스크립트 파싱을 통과했다.
- Stage102 Android 포커스 단위 검증: Android에서는 지연 스크롤이 예약되지 않고, 평가 동작 시 텍스트 포커스만 해제되며 일반 입력 포커스는 유지됨을 확인했다.
- Stage102 iOS 회귀: 포커스를 잃은 이전 입력칸의 예약 스크롤은 취소되고, 현재 포커스가 유지되는 iOS 입력칸만 중앙 정렬됨을 확인했다.
- Stage102 실제 모바일 화면: 360×800에서 MOB 선수 이름 입력 직후 화면 아래의 점수 슬라이더를 눌렀다. 520ms 뒤에도 활성 포커스와 화면은 점수 슬라이더에 유지됐고 이름 입력칸으로 돌아가지 않았다.
- Stage102 모바일 레이아웃·콘솔: 문서 폭 360px, 가로 넘침 없음, 브라우저 경고·오류 0건을 확인했다.
- Stage102 변경 격리: 평가 HTML은 모바일 포커스 블록만 Stage101과 다르고 나머지 전체 내용은 동일하다. 서버 API는 Stage101과 바이트 단위로 동일하며 6개 HTML 인라인 스크립트와 JavaScript 문법 검사를 통과했다.
- 운영 주소 확인: 2026-07-16 기준 `/`, `/assessment/`, `/api/health`가 모두 HTTP 200이다. Stage102 코드는 아직 운영 주소에 배포하지 않았다.
- 초기화 격리: `ALL` 초기화 거부, MOB 2건 초기화 후 KBC 1건 유지 확인.
- 미검수·켈리브레이션: 순위와 최종정리에서 제외, 백업 데이터에서는 구분 보존됨.
- 미검증: 원격 Cloudflare D1, 실제 전체 등록 엑셀, 실제 SOLAPI 문자, 모든 실기기 카메라/인쇄 조합.
